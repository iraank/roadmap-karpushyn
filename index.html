01

Продукт

Продукт

The product

AI-коуч підтримки між сесіями. Не чат-порадник, а дзеркало: він не відповідає на питання, а ставить їх — так, як це роблю я в коучингу. Клієнт працює з ним щодня між живими сесіями зі мною.

Бізнес-логіка. AI — не кінцевий продукт, а вхідна воронка в живий коучинг. Людина спілкується з AI, на певній глибині впирається в межу («а чому саме Я?») — і сама хоче живу зустріч. Дешева підписка як вхід у дорогий контракт.

Найближча мета. Довести на 10–30 своїх клієнтах, що це працює (закрита бета, 3 місяці). Не «платформа для всіх» — одна задача.

Мова. Застосунок — для українців будь-якою мовою (діаспора ЄС думає місцевою: чеська, польська, німецька…). Клієнт обирає мову на вході, і коуч говорить нею.

🏛️ Візуальна схема застосунку — без коду

AI-коуч поддержки между сессиями. Не чат-советчик, а зеркало: он не отвечает на вопросы, а задаёт их — так, как это делаю я в коучинге. Клиент работает с ним каждый день между живыми сессиями со мной.

Бизнес-логика. AI — не конечный продукт, а входная воронка в живой коучинг. Человек общается с AI, на определённой глубине упирается в границу («а почему именно Я?») — и сам хочет живую встречу. Дешёвая подписка как вход в дорогой контракт.

Ближайшая цель. Доказать на 10–30 своих клиентах, что это работает (закрытая бета, 3 месяца). Не «платформа для всех» — одна задача.

Язык. Приложение — для украинцев на любом языке (диаспора ЕС думает на местном: чешский, польский, немецкий…). Клиент выбирает язык на входе, и коуч говорит на нём.

🏛️ Визуальная схема приложения — без кода

An AI coach for support between sessions. Not a chat-advisor but a mirror: it doesn't answer questions, it asks them — the way I do in coaching. The client works with it daily between our live sessions.

Business logic. The AI isn't the end product — it's the top-of-funnel into live coaching. A person talks to the AI, at a certain depth hits a wall ("but why me?") — and wants a live session themselves. A cheap subscription as the entry to a high-ticket contract.

Immediate goal. Prove it works on 10–30 of my own clients (closed beta, 3 months). Not a "platform for everyone" — one job.

Language. The app is for Ukrainians in any language (the EU diaspora often thinks in the local one: Czech, Polish, German…). The client picks a language at the entrance and the coach speaks it.

Стек

Стек

The stack

Форма
Веб-застосунок (телефон + десктоп), не Telegram-бот. Стиль навчального продукту.
Стек
Next.js (фронт) + Supabase (база, реєстрація, пам'ять) + Claude API (модель, без fine-tuning — уся «особистість» у системному промпті). Хостинг Vercel.
Пам'ять
Живе зовні моделі: у промпт іде не вся історія чату, а структурований стан клієнта («жива анкета»: точка А, теми, етап) + резюме. Щоб не роздувати контекст і тримати якість.
Два входи
Клієнтський застосунок + окрема консоль для коуча (бачу своїх людей, додаю контекст живих сесій).
Мови
Застосунок для українців будь-якою мовою. Клієнт обирає мову на вході; коуч говорить нею — Claude багатомовний нативно, без окремого перекладача. Інтерфейс — легкий словник рядків. Дешево, масштабується.
Можливий ризик
Ізоляція даних між клієнтами (RLS) — клієнт А не має бачити клієнта Б. Хочу, щоб досвідчена людина це перевірила перед реальними людьми.
Спосіб
Збірка вайбкодингом (я не програміст), розкладена на 7 етапів від голосу до бети.
Форма
Веб-приложение (телефон + десктоп), не Telegram-бот. Стиль обучающего продукта.
Стек
Next.js (фронт) + Supabase (база, регистрация, память) + Claude API (модель, без fine-tuning — вся «личность» в системном промпте). Хостинг Vercel.
Память
Живёт вне модели: в промпт идёт не вся история чата, а структурированное состояние клиента («живая анкета»: точка А, темы, этап) + резюме. Чтобы не раздувать контекст и держать качество.
Два входа
Клиентское приложение + отдельная консоль для коуча (вижу своих людей, добавляю контекст живых сессий).
Языки
Приложение для украинцев на любом языке. Клиент выбирает язык на входе; коуч говорит на нём — Claude многоязычен нативно, без отдельного переводчика. Интерфейс — лёгкий словарь строк. Дёшево, масштабируется.
Возможный риск
Изоляция данных между клиентами (RLS) — клиент А не должен видеть клиента Б. Хочу, чтобы опытный человек это проверил перед реальными людьми.
Способ
Сборка вайбкодингом (я не программист), разложена на 7 этапов от голоса до беты.
Form
Web app (phone + desktop), not a Telegram bot. In the style of a learning product.
Stack
Next.js (frontend) + Supabase (database, auth, memory) + Claude API (model, no fine-tuning — the whole "personality" lives in the system prompt). Hosted on Vercel.
Memory
Lives outside the model: the prompt gets not the whole chat history but a structured client state (a "living profile": point A, themes, stage) + summaries. To keep context small and quality high.
Two entrances
A client app + a separate coach console (I see my people, add context from live sessions).
Languages
The app is for Ukrainians in any language. The client picks a language at the entrance; the coach speaks it — Claude is natively multilingual, no separate translator. The interface uses a lightweight string dictionary. Cheap, scales.
A possible risk
Data isolation between clients (RLS) — client A must never see client B. I want an experienced person to check this before real users.
Approach
Built by vibe-coding (I'm not a programmer), broken into 7 stages from voice to beta.

Етапи роботи

Этапы работы

Work stages

  • П’ять глибоких досліджень — ринок, техстек, техархітектура конкурентів, Product Hunt (198 конкурентів), гранти (таблиця нижче). Рішення ухвалені, не гадаю на ходу.
  • Робочий прототип V1 — чат із Claude українською в режимі підтримки + захисний SAFE PATH (стоп-скринінг кризових сигналів → переадресація до фахівця). Перевірено.
  • Цифровий голос — мій Tone of Voice (верифікований документ) вшитий у системний промпт: правила, лексика, табу, приклади.
  • Пам'ять працює — Supabase підключено (клієнти, повідомлення, «жива анкета», прапорці, RLS), реєстрація і історія розмов пишуться в базу. Ключі Anthropic + Supabase живі, перевірені.
  • Інфраструктура — приватні репозиторії з ліцензією «всі права застережено», повний план збірки (8 етапів) з критеріями готовності + Zettelkasten-база знань (university/) з простежуваністю кожного рішення до джерела.
  • Пять глубоких исследований — рынок, техстек, техархитектура конкурентов, Product Hunt (198 конкурентов), гранты (таблица ниже). Решения приняты, не гадаю на ходу.
  • Рабочий прототип V1 — чат с Claude на украинском в режиме поддержки + защитный SAFE PATH (стоп-скрининг кризисных сигналов → переадресация к специалисту). Проверено.
  • Цифровой голос — мой Tone of Voice (верифицированный документ) вшит в системный промпт: правила, лексика, табу, примеры.
  • Память работает — Supabase подключён (клиенты, сообщения, «живая анкета», флаги, RLS), регистрация и история разговоров пишутся в базу. Ключи Anthropic + Supabase живые, проверены.
  • Инфраструктура — приватные репозитории с лицензией «все права защищены», полный план сборки (8 этапов) с критериями готовности + Zettelkasten-база знаний (university/) с прослеживаемостью каждого решения до источника.
  • Five deep-research reports — the market, the tech stack, competitors' tech architecture, Product Hunt (198 competitors), grants (table below). Decisions are made, not guessed on the fly.
  • A working V1 prototype — a Claude chat in Ukrainian in support mode + a protective SAFE PATH (crisis-signal stop-screening → referral to a specialist). Tested.
  • A digital voice — my Tone of Voice (a verified document) embedded in the system prompt: rules, vocabulary, taboos, examples.
  • Memory works — Supabase is wired up (clients, messages, "living profile", flags, RLS); registration and conversation history are written to the DB. Anthropic + Supabase keys are live and verified.
  • Infrastructure — private repositories with an "All Rights Reserved" license, a full build plan (8 stages) with done-criteria + a Zettelkasten knowledge base (university/) tracing every decision to its source.

Питання на зустріч

Вопросы на встречу

Questions for the meeting

Блок A — Продукт і архітектура

  • Як робив би ти AI-коуча мислення? З чого би почав і що робив би інакше?
  • Ізоляція даних між клієнтами (RLS) — це справді складний етап, і мені здається, що тут потрібен разовий аудит безпеки перед бетою. Чи це перебільшення?
  • Рішення про майбутнє — на валідацію. Попереду ймовірні розширення: школа на методі Ісаєва (інші коучі), згодом — моя власна методологія Life OS. Я вирішила закладати мінімально: одна колонка готовності в схемі, решта — коли з'явиться реальний тригер. Бачиш у цьому проблему?
  • Спрощення воронки. Вхід максимально простий: тільки email + 7 днів безкоштовного спілкування. Чи можна ще простіше?

Блок B — Менторинг

  • LISPR. Я в першій сотні користувачів. Чула від тебе, що вийти на високий рівень достатньо складно — чи не міг би ти трошки більше про це розповісти?
  • Мультимовність. Мене надихнуло, що LISPR перекладає в режимі реального часу — і я подумала: а чому б і мені так не зробити? Застосунок — для українців будь-якою мовою. План: коуч говорить обраною мовою (Claude нативно), інтерфейс — легкий словник рядків. Це правильний підхід — чи ти зробив би інакше?
  • Я роблю новий продукт, не отримавши гроші в попередньому. Я зробила воронку — лендінг + Telegram-бот, довела її до робочого стану — і спинилась, не дійшовши до живих грошей. Зараз з азартом розпочала AI Coach, який спочатку теж виглядає як воронка. Поки готувалася до зустрічі з тобою, усвідомила, що мені цікаво створювати продукти, але я не довожу їх до реальних грошей. Ось моя лендінг-воронка з ТГ-ботом:
    ☕ Лендінг🐇 Бот воронки

Блок C — Навчання

  • Що читати і на що звертати увагу? Я прийшла на Ковчег з нуля, без технічних навичок — і зараз бачу, що їх усе ж не вистачає. Що б ти порадив читати? На що звертати увагу? І як робити правильніше — щоб не плодити зоопарк скілів?

Блок D — Інвестиції

  • Чи є сенс шукати інвестора? Чи міг би цей проєкт бути цікавим інвестиціям — на твою думку? І якщо ні — що в ньому змінити, щоб став цікавим?

Блок A — Продукт и архитектура

  • Как делал бы ты AI-коуча мышления? С чего бы начал и что делал бы иначе?
  • Изоляция данных между клиентами (RLS) — это действительно сложный этап, и мне кажется, что здесь нужен разовый аудит безопасности перед бетой. Или это преувеличение?
  • Решение о будущем — на валидацию. Впереди вероятные расширения: школа на методе Исаева (другие коучи), позже — моя собственная методология Life OS. Я решила закладывать минимально: одна колонка готовности в схеме, остальное — когда появится реальный триггер. Видишь в этом проблему?
  • Упрощение воронки. Вход максимально простой: только email + 7 дней бесплатного общения. Можно ли ещё проще?

Блок B — Менторинг

  • LISPR. Я в первой сотне пользователей. Слышала от тебя, что выйти на высокий уровень достаточно сложно — не мог бы ты чуть больше об этом рассказать?
  • Мультиязычность. Меня вдохновило, что LISPR переводит в реальном времени — и я подумала: а почему бы и мне так не сделать? Приложение — для украинцев на любом языке. План: коуч говорит на выбранном языке (Claude нативно), интерфейс — лёгкий словарь строк. Это правильный подход — или ты сделал бы иначе?
  • Я делаю новый продукт, не получив деньги в предыдущем. Я сделала воронку — лендинг + Telegram-бот, довела её до рабочего состояния — и остановилась, не дойдя до живых денег. Сейчас с азартом начала AI Coach, который поначалу тоже выглядит как воронка. Пока готовилась к встрече с тобой, осознала, что мне интересно создавать продукты, но я не довожу их до реальных денег. Блок C — Обучение

  • Что читать и на что обращать внимание? Я пришла на Ковчег с нуля, без технических навыков — и сейчас вижу, что их всё же не хватает. Что бы ты посоветовал читать? На что обращать внимание? И как делать правильнее — чтобы не плодить зоопарк скиллов?

Блок D — Инвестиции

  • Есть ли смысл искать инвестора? Мог бы этот проект быть интересным инвестициям — на твой взгляд? И если нет — что в нём изменить, чтобы стал интересным?

Block A — Product & architecture

  • How would you build an AI thinking coach? Where would you start, and what would you do differently?
  • Client data isolation (RLS) — this really is a hard part, and it seems to me a one-off security audit is needed before the beta. Or is that an overreaction?
  • A decision about the future — for validation. Likely extensions ahead: a school on Isaiev's method (other coaches), later — my own methodology, Life OS. I decided to build in the minimum: one readiness column in the schema, the rest — when a real trigger appears. Do you see a problem with that?
  • Simplifying the funnel. The simplest possible entry: email only + 7 days of free conversation. Can it be even simpler?

Block B — Mentoring

  • LISPR. I'm among its first hundred users. I've heard from you that getting to a high level is quite hard — could you tell a bit more about that?
  • Multilingual. I was inspired that LISPR translates in real time — and I thought: why not do the same? The app is for Ukrainians in any language. The plan: the coach speaks the chosen language (Claude natively), the interface uses a lightweight string dictionary. Is this the right approach — or would you do it differently?
  • I'm building a new product without having earned money on the previous one. I built a funnel — a landing page + Telegram bot, got it to a working state — and stopped, without ever reaching real money. Now I've eagerly started AI Coach, which at first also looks like a funnel. While preparing for this meeting, I realized I love creating products, but I don't take them through to real money. Block C — Learning

  • What should I read and pay attention to? I came to Kovcheg from zero, with no technical skills — and I can see now that they're still lacking. What would you recommend reading? What should I pay attention to? And how do I do this properly — without breeding a zoo of skills?

Block D — Investment

  • Is it worth looking for an investor? Could this project be interesting to investors — in your view? And if not — what would need to change for it to become interesting?
